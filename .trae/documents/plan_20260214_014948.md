# 解决 Memos 在内网穿透/代理环境下的认证问题

**问题分析**
您遇到的 `refresh token not found` 错误通常是因为浏览器没有发送 `memos_refresh-token` Cookie 给后端。在通过 DDNSto 等内网穿透工具访问时（HTTPS），如果后端没有正确识别出当前是 HTTPS 环境，可能会导致 Cookie 属性（如 `Secure`）设置不正确，或者因为代理转发头丢失导致判断失误。

经过代码审查，发现 Memos 存在两个关键问题导致无法正确处理代理后的 HTTPS 请求：
1. **元数据拦截器缺失关键头**: `server/router/api/v1/connect_interceptors.go` 中的拦截器只转发了 `X-Forwarded-For`，但**忽略**了 `X-Forwarded-Proto`（用于标识协议是 HTTP 还是 HTTPS）。
2. **Cookie 生成逻辑不完善**: `server/router/api/v1/auth_service.go` 中的 `buildRefreshTokenCookie` 仅依赖 `Origin` 头来判断 HTTPS，未检查标准的代理头 `X-Forwarded-Proto`。

**解决方案**
我将通过修改代码来增强 Memos 对代理环境的支持：

1.  **修改 `connect_interceptors.go`**:
    -   在 `MetadataInterceptor` 中增加对 `X-Forwarded-Proto` 请求头的提取和转发，使其能传递给后续的业务逻辑。

2.  **修改 `auth_service.go`**:
    -   更新 `buildRefreshTokenCookie` 方法，在判断是否为 HTTPS 时，优先检查 `X-Forwarded-Proto` 头。
    -   这将确保即使在反向代理（如 DDNSto）剥离了 `Origin` 或通过 HTTP 与后端通信的情况下，Memos 也能正确识别 HTTPS 环境并设置 `Secure` Cookie。

**实施步骤**
1.  编辑 `server/router/api/v1/connect_interceptors.go`，添加 `X-Forwarded-Proto` 的处理。
2.  编辑 `server/router/api/v1/auth_service.go`，在 `isHTTPS` 判断逻辑中加入对 `x-forwarded-proto` 的检查。
